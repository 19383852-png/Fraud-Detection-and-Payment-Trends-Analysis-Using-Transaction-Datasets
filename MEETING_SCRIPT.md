Hello Professor, today I would like to briefly walk you through the baseline stage of my fraud-detection project. The main aim of this project is to analyse credit-card transaction patterns and build machine-learning models that can detect fraudulent transactions in a highly imbalanced dataset. I structured the work as a reproducible pipeline rather than keeping everything only inside notebooks, so each step can be reused, explained, and extended more easily. (Open [README.md](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/README.md#L43), lines 43-66)

To support that workflow, I organised the repository into separate areas for data, notebooks, source code, and outputs. The key idea was to move from exploratory analysis into a cleaner software structure where loading, preprocessing, modelling, and evaluation are separated into different modules. This makes the project easier to maintain and also easier to present, because each file has a clear responsibility. (Open [README.md](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/README.md#L9), lines 9-37)

I first created a shared configuration file so the pipeline is reproducible. In this file, I centralised the dataset path, fixed the random seed, and defined the test-set size. This means that when I rerun the code, I can reproduce the same split and obtain consistent experimental settings. (Open [src/config.py](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/src/config.py#L1), lines 1-10)

After that, I built a dedicated data-loading module. Instead of reading the CSV ad hoc in multiple places, I used one function to load the dataset and validate that the target column, called `Class`, is present. This helps ensure that the later modelling stage always starts from a consistent and verified dataset. (Open [src/data_loader.py](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/src/data_loader.py#L1), lines 1-14)

For preprocessing, I made a deliberate choice based on the dataset structure. The `Time` and `Amount` columns are raw numeric features, so I applied median imputation and standard scaling to them. However, the `V1` to `V28` variables are already PCA-transformed features, so I left them unchanged. This preserves the original meaning of the anonymised feature space while still standardising the raw variables that can vary widely in scale. (Open [src/preprocessing.py](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/src/preprocessing.py#L1), lines 1-22)

I then implemented two baseline models: Logistic Regression and Random Forest. Logistic Regression serves as an interpretable linear baseline, while Random Forest allows me to capture more complex non-linear patterns in the transaction data. In both cases, I added class weighting because the fraud class is extremely rare, and I wanted the models to pay more attention to minority-class detection. (Open [src/models.py](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/src/models.py#L1), lines 1-25)

The main orchestration happens in the baseline pipeline. Here, I load the data, split the predictors and target, and then perform a stratified train-test split. Stratification is important in this project because it preserves the original fraud ratio in both sets, which makes the evaluation more realistic. I also wrapped preprocessing and modelling inside scikit-learn pipelines, which prevents data leakage by ensuring that transformations are learned only from the training data and then consistently applied to the test data. (Open [src/baseline.py](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/src/baseline.py#L15), lines 15-41)

I also kept the pipeline extensible by including an optional SMOTE branch. The reason for that is that this baseline stage establishes the natural, imbalanced benchmark first, and then later I can compare it with an oversampled training workflow. So the same pipeline can support both the baseline experiment and the next step of the project. (Open [src/baseline.py](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/src/baseline.py#L32), lines 32-41)

For evaluation, I focused on metrics that are suitable for imbalanced classification. I used ROC-AUC and PR-AUC to assess ranking performance, and I also reported precision, recall, and F1-score through the classification report. I did this because accuracy alone would be misleading in a dataset where the vast majority of transactions are legitimate. (Open [src/evaluate.py](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/src/evaluate.py#L1), lines 1-13)

When I performed the stratified split, the class imbalance was preserved very closely. The training set contained 241,667 legitimate transactions and 418 fraud cases, while the test set contained 42,648 legitimate transactions and 74 fraud cases. This confirms that the split reflects the real distribution of the dataset and supports a fair evaluation setup. (Open [outputs/metrics/data_split.txt](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/outputs/metrics/data_split.txt#L1), lines 1-14)

The Logistic Regression baseline produced a very high fraud recall of 0.9189, which means it successfully identified most of the fraud cases. However, its precision for fraud was only 0.0656, so it generated many false positives. In practice, this means it is very sensitive, but it would likely create too many false alerts in a real payment system. (Open [outputs/metrics/logreg_baseline.txt](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/outputs/metrics/logreg_baseline.txt#L4), lines 4-14)

The Random Forest baseline showed a much stronger balance between sensitivity and precision. Its fraud precision was 0.8615, recall was 0.7568, and the F1-score was 0.8058. This means it missed more fraud cases than Logistic Regression, but when it predicted fraud, it was much more reliable. For that reason, I consider Random Forest the stronger practical baseline in this stage of the project. (Open [outputs/metrics/rf_baseline.txt](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/outputs/metrics/rf_baseline.txt#L12), lines 12-22)

The comparison table summarises this trade-off clearly. Logistic Regression achieved stronger recall, while Random Forest achieved a much better balance across precision, recall, and F1-score. So the key takeaway from this stage is not simply which model is better overall, but how the choice depends on whether the priority is catching as many fraud cases as possible or reducing the operational cost of false positives. (Open [outputs/tables/baseline_results.csv](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/outputs/tables/baseline_results.csv#L1), lines 1-3)

In summary, this stage of the project helped me establish a clean and reproducible fraud-detection baseline, justify my preprocessing choices, compare two different modelling strategies, and identify the trade-off between recall and precision in imbalanced classification. This also sets up the next phase of the project, where I will investigate methods such as SMOTE to address the class imbalance more explicitly. ## Likely Questions

Why did you use stratified splitting?
Because fraud is extremely rare, and a normal random split could distort the class ratio between training and test data. Stratification keeps the evaluation realistic. (Open [src/baseline.py](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/src/baseline.py#L23), lines 23-27)

Why did you not scale the `V1` to `V28` columns?
Those variables are already PCA-like transformed features, so scaling them again was not necessary for the baseline pipeline. (Open [src/preprocessing.py](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/src/preprocessing.py#L8), lines 8-10)

Why did you use PR-AUC as well as ROC-AUC?
PR-AUC is more informative when the positive class is very rare, because it focuses more directly on fraud-detection quality. (Open [src/evaluate.py](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/src/evaluate.py#L5), lines 5-13)

Why is Logistic Regression not enough on its own?
It gives very high recall, but the precision is too low, which means too many legitimate transactions would be incorrectly flagged as fraud. (Open [outputs/metrics/logreg_baseline.txt](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/outputs/metrics/logreg_baseline.txt#L9), lines 9-14)

Why is Random Forest a better practical baseline?
Because it offers a much stronger balance between catching fraud and avoiding false positives, which is reflected in its much higher fraud F1-score. (Open [outputs/metrics/rf_baseline.txt](/Users/zhangshuai/Desktop/Projects/Fraud-Detection-and-Payment-Trends-Analysis-Using-Transaction-Datasets/outputs/metrics/rf_baseline.txt#L17), lines 17-22)
